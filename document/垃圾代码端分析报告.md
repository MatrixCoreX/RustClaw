# 反复修改导致的垃圾代码端分析报告

更新时间：2026-03-17

## 结论概览

| 端 | 主要问题 | 与 memory-usage-plan 关系 |
|----|----------|---------------------------|
| **clawd/main.rs** | 单文件过大(1万行+)、重复的「lock→检查→return (StatusCode, Json(ApiResponse))」模式、RateLimiter 未清理 per_user 空键 | Phase 1：限流 map 清理 |
| **clawd/agent_engine.rs** | 每轮都重新拼接技能 playbook 大字符串、format!/push_str 等 151 处 | Phase 2：playbook 缓存 |
| **telegramd/main.rs** | 单文件 5294 行、多处「if let Ok(mut guard) = state.xxx.lock()」分散、三个运行态 HashMap 无 TTL 清扫 | Phase 1：pending map TTL |
| **crypto-skill** | 单文件 4319 行、多交易所逻辑重复（binance/okx 等）、配置与请求路径重复 | 无直接关系，属结构债 |

---

## 1. clawd/src/main.rs

### 1.1 文件过大与重复响应模式

- **行数**：约 10055 行，单文件承担启动、路由、限流、任务提交、维护、UI 等，可读性与 MR 冲突成本高。
- **重复模式**：`Json(ApiResponse { ... })` 出现约 35 处，多为：
  - 先 `lock()`，`Err` 时 `return (StatusCode::INTERNAL_SERVER_ERROR, Json(ApiResponse { ok: false, ... }))`
  - 再业务检查失败时 `return (StatusCode::TOO_MANY_REQUESTS/..., Json(ApiResponse { ... }))`
- **建议**：
  - 抽成辅助函数，例如 `fn api_err(status: StatusCode, message: &str) -> (StatusCode, Json<ApiResponse>)`，或 `api_ok(data)`，减少复制粘贴与不一致。
  - 中长期：按职责拆成模块（如 `submit.rs`、`limits.rs`、`maintenance.rs`），main 只做组装与路由。

### 1.2 RateLimiter：per_user 只增不减（已知可累积点）

- **位置**：约 685–719 行，`RateLimiter::check_and_record`。
- **现状**：只对每个 `user_id` 的 `VecDeque` 做时间窗口内 pop_front，**从不对 `per_user` 做删除**。用户一旦出现过就会永久占一条键，长期运行键数量只增不减。
- **建议**（与 memory-usage-plan Phase 1 一致）：
  - 在 `check_and_record` 里，对当前 `user_id` 清理完过期时间戳后，若 `user_q.is_empty()`，则从 `per_user` 中 `remove(user_id)`。
  - 可选：定期对 `per_user` 做一次「清理空 VecDeque 的键」的 retain，避免只依赖“下次该用户再请求时才清理”的延迟。

## 2. clawd/src/agent_engine.rs

### 2.1 每轮都拼接技能 playbook，无缓存

- **位置**：`build_skill_playbooks_text()`（约 51–116 行），在 `run_agent_with_tools` 路径下约 1306 行被调用。
- **现状**：每次 ask 都会：
  - 取 `planner_visible_skills_for_task(task)`
  - 对每个技能读 registry、`load_prompt_template_for_state`、拼接成一大段 `String`
  - 无按「技能集 + 配置版本」的缓存，内存与 CPU 每轮都重复峰值。
- **建议**（与 memory-usage-plan Phase 2 一致）：
  - 对「技能名集合 + 相关配置版本/哈希」做一层 playbook 文本缓存（例如 `Arc<str>` 或 `String`），仅在配置/技能列表变化时失效。
  - 可先做「按 agent_id + skills_list 的哈希」的简单缓存，再考虑与 `skill_views_snapshot` 的失效联动。

### 2.2 字符串拼接过多

- **现状**：该文件中 `format!` / `push_str` / `String::from` / `to_string` 等约 151 处，大量用于构造 prompt、日志、错误信息。
- **建议**：在 playbook 缓存落地后，再评估哪些是「每轮必建」、哪些可复用；优先减少大块 prompt 的重复分配，其次再考虑小字符串。

## 3. telegramd/src/main.rs

### 3.1 单文件过大与锁使用分散

- **行数**：约 5294 行，所有消息处理、回调、发图、语音、crypto 确认等都在同一文件。
- **锁模式**：`state.pending_resume_by_chat`、`pending_image_by_chat`、`bound_identity_by_chat`、`pending_key_bind_by_chat` 等，多处 `if let Ok(mut guard) = state.xxx.lock() { ... }` 分散在不同行（如 199、439、664、1728、2075、3938、4813 等），风格不统一（有的 `map_err` 提前 return，有的忽略 `Err`）。
- **建议**：
  - 为「获取并可选修改」封装小函数，例如 `with_pending_resume<R>(state, chat_id, f: impl FnOnce(&mut HashMap<...>) -> R) -> Option<R>`，统一 lock 与错误处理，避免散布的 lock 与不一致的 poison 处理。
  - 中长期：按「消息类型/流程」拆文件（如 `resume.rs`、`image.rs`、`crypto_callback.rs`），便于加 TTL 和测试。

### 3.2 三个运行态 HashMap 无 TTL（已知可累积点）

- **位置**：
  - `pending_resume_by_chat`（约 65、439、3938、4813 等）
  - `pending_image_by_chat`（约 48、664、1728、2075 等）
  - `bound_identity_by_chat`（约 67、199、4275 等）
- **现状**：只有「用完后 remove」或「校验失败时 remove」，没有「按时间过期」的清扫，chat 数量多且长期运行会缓慢增长。
- **建议**（与 memory-usage-plan Phase 1 一致）：
  - 为每个 map 的 value 增加或复用「最后活跃时间」字段。
  - 在合适入口（如定时任务或某类消息处理前）做 TTL 清扫：删除「超过 N 分钟/小时未活跃」的条目，注意不要打断进行中的确认/恢复流程（例如只删已过期的，且不与正在用的 key 冲突）。

## 4. crates/skills/crypto/src/main.rs

### 4.1 单文件过大与多交易所重复

- **行数**：约 4319 行，配置解析、Binance/OKX 等请求、订单/持仓/余额等逻辑全在一个文件。
- **重复模式**：各交易所的「请求签名 → 发 HTTP → 解析」类似，但各自实现（如 `binance_signed_request`、`okx_request` 等），导致：
  - 相似错误处理、重试、日志多处复制；
  - 新增交易所时继续复制粘贴，易产生不一致和遗漏。
- **建议**：
  - 抽象一层「交易所客户端」trait 或模块：统一「签名 + 请求 + 解析」接口，各交易所实现一个薄适配层。
  - 将「action 分发」与「各 action 的实现」拆到不同模块（如 `actions/positions.rs`、`actions/order.rs`、`exchanges/binance.rs`、`exchanges/okx.rs`），便于复用和测试。
  - 配置默认值（如各种 `default_*_api_path`）可集中到一个 `defaults` 模块，减少重复的 `#[serde(default = "default_xxx")]` 与定义分散。

### 4.2 与「反复改」直接相关的点

- **LegacyRootConfig / RootConfig**：两套配置结构并存，说明历史上做过兼容旧配置的改动，可考虑在文档中标明废弃时间，并在新版本中收敛到单一结构。
- **注释中的 `#[allow(dead_code)]`（如 require_explicit_send）：「保留配置兼容」会导致字段和分支长期存在，建议在配置文档中注明「已废弃」，避免后续再为兼容性加分支。

## 5. 建议的优先顺序（与 memory-usage-plan 对齐）

1. **先做 Phase 1（低风险、可观测）**
   - clawd：`RateLimiter.per_user` 在清理过期时间戳后删除空队列的键。
   - telegramd：为 `pending_resume_by_chat`、`pending_image_by_chat`、`bound_identity_by_chat` 增加 TTL 清扫逻辑。
2. **再减重复与体积**
   - clawd/main.rs：抽出 `api_err`/`api_ok` 等统一响应构造，减少 35 处重复。
   - agent_engine：为 `build_skill_playbooks_text` 增加按技能集/配置的缓存（Phase 2）。
3. **结构性拆分（按需、分 PR）**
   - clawd：将 submit、limits、maintenance 等拆成子模块。
   - telegramd：按流程拆出 resume/image/crypto 等模块，并统一 lock 封装。
   - crypto-skill：交易所抽象 + action 与 exchange 分模块。

## 6. 验收自检（垃圾代码治理后）

- [ ] `RateLimiter.per_user.len()` 在长期运行下会回落（有清理）。
- [ ] telegramd 运行态 map 大小随活跃 chat 波动，不单调增长。
- [ ] clawd 提交任务相关代码中，错误返回路径使用统一辅助函数，无新增手写 `Json(ApiResponse { ... })` 复制粘贴。
- [ ] 技能 playbook 在未改配置/技能列表时，不每轮重新拼接大字符串（有缓存或等效优化）。
- [ ] 新改动不增加单文件行数（优先在现有大文件中用「抽函数/抽模块」替代「再写一段」）。

上述项可作为 PR 自检或 code review 的检查点，避免「反复改」继续堆积在同一位置。
