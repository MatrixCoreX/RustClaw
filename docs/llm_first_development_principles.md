# RustClaw 开发宗旨（LLM 优先）

## 目标
RustClaw 的核心目标是充分发挥大模型在语义理解、任务分解、工具编排和多轮上下文处理上的优势。  
系统应优先把“能力边界 + 可用工具 + 当前上下文”提供给模型，由模型做决策与编排，而不是靠大量硬编码分支驱动。

## 核心原则
1. **LLM 优先编排**  
优先让模型决定“做什么、按什么顺序做、何时澄清、何时执行”，代码侧只做协议、权限、超时和安全兜底。

2. **能力显式告知**  
要清楚告诉模型：有哪些 skill/tool、每个能力能做什么、参数契约是什么、失败会返回什么。  
模型不知道的能力，不应期待它稳定调用。

3. **优先准确完成，而不是节省 token**  
在当前产品阶段，token 消耗不是首要优化目标。  
只要能明显提升一次完成率、减少补问轮次、降低误判和误执行，就应优先给模型足够上下文、能力说明和约束信息。  
不要为了省 token 过早裁剪关键上下文、拆碎请求、删掉必要约束，导致模型需要更多轮交互才能完成同一个目标。

4. **减少硬匹配**  
避免通过关键词、固定短语、语言特定表达做路径分流。  
优先使用模型语义判断与上下文推理，降低多语言场景维护成本。

5. **允许逻辑规则，但不替代语义理解**  
规则用于“安全、合规、限流、幂等、协议校验”等确定性场景。  
规则不应过度侵入意图理解与任务规划。

6. **先澄清再执行**  
目标不唯一、参数缺失、上下文冲突时，优先让模型发起澄清问题，不做静默猜测执行。

7. **上下文是证据，不是命令**  
recent execution、memory、binding 仅作为候选证据。  
是否采纳由模型结合当前请求判断，不做机械复用。

8. **多语言一致性**  
输出语言应与用户语言一致或兼容，避免写死中文/英文文案逻辑。

9. **可观测与可回归**  
关键判断（路由、澄清、技能调用）要可记录、可复盘、可通过 NL 回归集验证。

## 工程落地要求
1. 提示词必须覆盖：角色、目标、输入上下文、输出格式、失败策略、澄清策略。  
2. skill 文档必须声明接口契约（action/参数/示例/错误）。  
3. 代码只做最小必要确定性控制：鉴权、白名单、超时、重试、JSON 协议校验。  
4. 不新增与语言绑定的硬编码分支（如固定中文词段触发）。  
5. 不为节省 token 牺牲完成率；能减少补问和误判的上下文、契约、示例、边界说明，应优先保留。  
6. 新逻辑必须附带回归用例，覆盖至少：成功、澄清、失败三类路径。

## 跨平台实现原则
1. **Rust 层优先使用编译期分平台实现**  
涉及进程探测、系统指标、端口扫描、服务管理、文件系统差异、系统命令差异等平台相关逻辑时，优先使用 `#[cfg(target_os = "...")]`、平台模块拆分或平台辅助函数，在编译期选择实现。不要把多平台核心逻辑长期堆在同一个函数里靠大量 `if os == ...` 运行时分支硬切。

2. **默认按“macOS / Linux”分层，而不是按发行版碎片化**  
Rust 主实现优先区分 `macOS` 与 `Linux`。`Ubuntu`、`Debian`、树莓派常见 Linux 发行版通常先复用同一套 Linux 路径，只有在系统调用、命令参数、打包形态确实不同的时候，才继续细分。

3. **把平台差异收敛到边界层，不扩散到业务层**  
平台判断应尽量放在 `system_health`、系统适配器、平台 helper、启动封装等边界模块。上层业务、路由、UI 接口、skill 协议应尽量消费统一语义结果，而不是到处传播平台判断。

4. **运行时分支只作为脚本层和外壳层补充**  
Shell、Node、部署脚本可以保留运行时平台检测，因为它们天然要处理包管理器、路径、服务管理器差异；但 Rust 主体中能用编译期分流的，优先不要退回成运行时分支。

5. **新增跨平台逻辑必须说明“为什么要分平台”**  
提交实现时，应能回答：平台差异点是什么、为什么不能共用一套实现、是否已经把差异收敛在最小范围内、主机平台默认编译和验证路径是什么。

6. **跨平台改动要至少验证当前主机平台，并尽量覆盖另一主平台**  
如果当前机器是 `macOS`，至少保证 `macOS` 路径编译/测试通过，并尽量补 `Linux` 侧的编译检查；如果当前机器是 `Linux`，则反过来执行。避免只改条件编译代码但完全不验证宿主路径。

## `docs/` 目录开发原则
1. **文档也是产品接口**  
`docs/` 不只是给开发者看的备忘录，也服务于后续 agent、评审者和维护者。  
文档应帮助读者快速理解“为什么这样设计、边界在哪里、改动会影响什么”。

2. **优先维护单一事实源**  
规范类、契约类、原则类内容应尽量落在稳定主文档中，避免同一规则散落在多份文档里。  
如果新增文档只是对已有规则换个说法，优先补充原文档而不是再开一份近似副本。

3. **区分规范文档与阶段性文档**  
长期有效的约束、契约、原则应写进稳定文件。  
阶段性方案、拆分计划、 rollout/checklist 可以单独建文档，但应明确它是阶段性产物，不要伪装成长期规范。

4. **优先解释约束背后的原因**  
不要只写“要这样做”，还要说明“为什么这样做、想避免什么问题、违背后会出现什么退化”。  
这样模型和人类维护者都更容易在新场景下做一致判断，而不是机械套规则。

5. **示例用于帮助理解，不应构成闭集词表**  
文档中的文件名、提示词片段、用户说法、技能名示例应明确是 example / illustrative。  
除非是协议字段、固定枚举或兼容性要求，不要把示例写成隐性硬匹配清单。

6. **明确区分语义原则与协议硬约束**  
“如何理解用户意图、如何判断目录语义、何时倾向澄清”这类内容应写成语义原则。  
`FILE:<path>`、JSON shape、action 参数、错误返回、扫描上限这类内容才应写成硬约束。

7. **文档结构优先面向任务与决策**  
优先按“目标、原则、边界、反模式、验收标准”来组织，而不是按代码文件顺序堆砌。  
让读者先知道该怎么判断，再知道具体实现落点。

8. **改代码时同步改文档，改文档时注明影响范围**  
涉及提示词分层、路由规则、skill 契约、文件发送、澄清机制等核心行为的改动，应同步更新 `docs/`。  
文档中最好说明影响模块、入口文件或逻辑路径，便于回归排查。

9. **减少“过期但仍像真的”文档**  
过时文档比没有文档更危险。  
若旧文档已不再代表当前行为，应及时合并、标注废弃、迁移链接或删除，避免 agent 和开发者读取后做出错误实现。

10. **文档应支持回归与复盘**  
对于容易退化的行为（如 clarify、delivery、路径定位、prompt layering），文档应尽量写清：
- 期望行为是什么
- 不应出现的反模式是什么
- 推荐用什么回归 case 或日志来验证

11. **对外表达保持克制、稳定、低门槛**  
面向产品说明或用户能力说明的文档，应遵循仓库里“非技术用户优先”的表达原则。  
少用炫技式术语，多用任务导向表达，例如“用户能做什么”“失败后下一步是什么”。

12. **允许版本化，但不要失去入口**  
当确实需要 `v1/v2` 方案文档时，应保留一个清晰入口文档说明当前推荐阅读顺序、各版本状态和适用范围。  
不要让 `docs/` 变成只能靠文件名猜测最新版本的目录。

## 路由收敛约束（动态澄清）
1. `needs_clarify` 只允许由统一语义层（intent normalizer）产出，不再叠加独立的“指代澄清 gate”二次判定。  
2. 禁止在代码侧维护指代词词表、关键词硬匹配或正则分流来强行触发澄清。  
3. 澄清问题生成可以读取候选上下文作为解释素材，但不得覆盖 `needs_clarify` 的主判定结果。  
4. 若后续要增强澄清准确性，优先升级提示词与上下文结构，不回退到词面规则。

## 反模式（应避免）
- 用字符串 contains/正则硬匹配替代语义判断。  
- 模型已具备足够上下文时仍重复打断式澄清。  
- 提示词未声明能力边界，导致模型“猜能力”。  
- 业务逻辑散落在多处硬编码，提示词与代码相互冲突。  
- 失败只报技术错误，不给用户下一步操作建议。

## 验收标准
一个改动若要被视为“符合 LLM 优先开发宗旨”，至少满足：
1. 模型能从提示词和技能文档中理解可用能力与约束。  
2. 主要决策由模型基于语义做出，而非关键词分支。  
3. 代码仅保留必要的确定性安全与协议兜底。  
4. 多语言输入下行为一致，不依赖单一语言词面。  
5. 有可复现实验或回归日志证明行为改进。
