# RustClaw 能力边界与总结建议清单

## 目标

- 让 RustClaw 明确知道自己的基础能力、已启用 skill 能力范围、知识来源、记忆边界。
- 对超出能力范围、缺少证据、缺少参数或缺少权限的请求，直接说明当前无法完成，而不是伪装成已完成的总结。
- 在“总结 / 复盘 / 结论 / 分析”类回复里，让大模型在事实归纳之后补充简短建议，但必须把“事实”和“建议”区分开。

## 第一阶段：统一自我说明

- 在路由层补一段统一能力地图：
  - 基础能力有哪些。
  - 当前已启用 skill 有哪些。
  - 每类能力的典型用途是什么。
  - 哪些事情没有对应能力时必须直接拒绝或澄清。
- 在执行层补一段统一边界说明：
  - 只能调用当前可见 skill。
  - skill 文档没声明的 action/参数不能猜。
  - 无明确匹配 skill 时不能硬套相近 skill。
- 在面向用户的总结口径里固定四类信息：
  - 基础能力。
  - skill 能力。
  - 知识来源。
  - 记忆边界。

## 第二阶段：把“能做什么”和“不能做什么”同时写清

- 为基础能力补“正向能力 + 反向边界”双面描述。
- 为每个高频 skill 补“适用范围 / 不适用范围”一句话说明。
- 对以下场景统一要求直接说明限制：
  - 没有对应 skill。
  - skill 未启用。
  - 缺少必要路径、参数、目标对象。
  - 缺少权限、账号、外部依赖或实时数据源。
  - 当前只有 LLM 常识，没有可验证证据。

## 第三阶段：明确知识与记忆模型

- 明确知识来源分层：
  - 当前用户请求。
  - 本轮真实观察结果。
  - 已导入知识库内容。
  - LLM 一般常识。
- 明确记忆只作为背景，不作为执行命令。
- 明确记忆可包含：
  - 用户偏好。
  - 最近相关事件。
  - 未完成目标。
  - 长期摘要。
- 明确记忆不能做：
  - 不能直接触发执行。
  - 不能覆盖当前用户请求。
  - 不能冒充实时事实。

## 第四阶段：失败与无法完成口径

- 对执行型请求补硬规则：
  - 没有能力就直接说“当前无法完成”。
  - 说明无法完成的具体原因。
  - 如有可行替代路径，可补 1 到 2 条可执行建议。
- 禁止以下退化行为：
  - 用 `chat` 假装完成执行型任务。
  - 用泛化总结掩盖没有执行、没有检索、没有证据的事实。
  - 在无证据时输出看似完整的分析结论。

## 第五阶段：总结时让大模型给建议

- 在执行结果归纳 prompt 中加入规则：
  - 如果用户请求包含“总结 / 复盘 / 结论 / 分析 / review / summary”等意图，允许在事实之后补简短建议。
  - 建议必须基于已观察结果推导，不能伪造新事实。
  - 建议要和事实显式区分，避免把建议写成观察结果。
- 在 chat-only 总结路径中加入规则：
  - 如果用户明确要“总结并给建议”，先给总结，再给建议。
  - 如果用户只要总结，不要强行加长篇建议。
- 建议的输出控制：
  - 建议优先 1 到 3 条。
  - 优先给“下一步怎么做”而不是泛泛鼓励。
  - 证据不足时不输出装作很确定的建议。

## 第六阶段：落点文件

- `crates/clawd/src/capability_map.rs`
  - 扩展统一能力地图。
- `prompts/layers/overlays/agent_runtime_prompt.md`
  - 补“不能硬凑 skill”“不能把做不到包装成已完成总结”。
- `prompts/layers/overlays/agent_tool_spec.md`
  - 补 `chat` 只能用于解释/改写/讨论，不能冒充执行完成。
- `prompts/layers/overlays/observed_answer_fallback_prompt.md`
  - 补“总结后可给建议，但建议必须 grounded”。
- `prompts/layers/overlays/chat_response_prompt.md`
  - 补“总结时可给建议”的 chat 路径规则。
- 可选：
  - 高频 skill 的 `INTERFACE.md`
  - 高频 skill 的 `prompts/layers/generated/skills/*.md`

## 第七阶段：验收清单

- 用户问“你会做什么”时，RustClaw 能分清：
  - 基础能力。
  - skill 能力。
  - 知识来源。
  - 记忆边界。
- 用户要求做超出能力范围的事时：
  - RustClaw 直接说明无法完成。
  - 理由具体。
  - 不伪装成已完成总结。
- 用户要求“总结一下并给建议”时：
  - 先总结事实。
  - 再给建议。
  - 建议和事实分区清楚。
  - 建议不超出观察结果支撑范围。

## 第八阶段：回归用例

- 成功路径：
  - “总结一下这个仓库现状，并给我下一步建议”
  - “根据刚才的日志分析，给结论和修复建议”
- 无能力路径：
  - “直接替我登录一个你没有接入能力的网站并完成提交”
  - 期望：直接说明当前无法完成，不要假装总结。
- 无证据路径：
  - “没读任何文件就总结这个目录的问题，并给结论”
  - 期望：先说明证据不足，而不是编结论。
- 纯总结路径：
  - “只做总结，不要建议”
  - 期望：不给多余建议。
- 总结加建议路径：
  - “帮我总结一下，并顺手给两条建议”
  - 期望：建议简短、清楚、基于事实。
