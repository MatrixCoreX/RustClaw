# RustClaw Skill Setup Guide

This document is for RustClaw self-questions such as ÔÇťhow do I enable this skillÔÇŁ, ÔÇťwhat does this skill needÔÇŁ, ÔÇťwhere should I bind the API keyÔÇŁ, or ÔÇťwhat config should be changedÔÇŁ. The goal is not fixed wording. The goal is to ground answers in the real runtime entry points that exist in this repository.

## Answering Pattern

1. Start with the real entry point. Say whether the setup lives in a config file, an environment variable, a local database/API, a local dependency, or a login/session state.
2. State the scope clearly. Distinguish machine-wide config, repo config, per-`user_key` binding, and per-session/channel login state.
3. If the user is only missing one thing, ask for it. Do not stop at abstract documentation when the next step is obvious. For ordinary parameters or paths, ask directly. For secrets, tokens, passwords, or API keys, prefer a dedicated command, UI, local config, or API path instead of asking the user to paste the sensitive value into ordinary chat.
4. Do not default to ÔÇťedit this manuallyÔÇŁ when RustClaw can already inspect or change the workspace through existing skills.
5. Do not confuse config policy files with credential storage. Most importantly for `crypto`: exchange credentials are normally stored per `user_key` in the local database, not by editing `configs/crypto.toml`.
6. Editing files under `configs/` is an admin-only capability. If the current task is not authenticated as admin, reply with a no-permission answer and do not attempt the change.

## Example Answer Shape

ÔÇť`crypto` exchange credentials are not normally edited in `configs/crypto.toml`. That file controls policy and limits. The actual Binance / OKX credentials for the current key are stored in the local database table `exchange_api_credentials`, and the normal binding path is Telegram `/cryptoapi set ...` or `POST /v1/auth/crypto-credentials`. To avoid sending secrets through ordinary chat, use the Telegram command directly. If you want, I can format the exact Binance or OKX command for you.ÔÇŁ

ÔÇťIf you want to modify an exchange API key, that means updating the current keyÔÇÖs credential record for that exchange, not editing a shared global config file. The direct path is to run `/cryptoapi set ...` again for your current key; it overwrites your own previous value.ÔÇŁ 

ÔÇťIf the user only says ÔÇśmodify the exchange API keyÔÇÖ but does not name the exchange, first check `crypto.execution_mode` / `crypto.default_exchange`. If a default exchange is configured, use it and give the matching `/cryptoapi set ...` command. Ask exactly one clarification only when no default exchange is configured.ÔÇŁ
