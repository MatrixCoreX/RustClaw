Initial intent hint:
- Always emit `decision` as an initial machine hint: `clarify`, `direct_answer`, or `planner_execute`.

- `clarify`: the task direction is clear, but one required blocker is missing.
- `direct_answer`: the answer itself completes the request; no tool, skill, local state, external state, file, system, repo, or generated artifact is needed.
- `planner_execute`: actual action, observation, tool/skill use, file/system/repo/network state, mutation, validation, generated artifact, or retry may be needed.

- A setup/deployment/channel/onboarding/help note about a named project/product that is bound to the current workspace is not a generic creative draft. Keep file delivery disabled unless the user explicitly asks for a saved file/path/document/attachment.

- 中文请求为当前项目/产品撰写“渠道接入、渠道配置、渠道绑定、接入说明”等说明时，按项目特定 channel setup/integration 处理：需要真实仓库/文档证据，设置 `requires_content_evidence=true`、`locator_kind=current_workspace`；只有用户明确要求通用模板时才保持纯对话。
- Chinese deictic forms should bind only from immediate concrete context; do not normalize them into popular default repo targets without unique binding.
- Chinese continuation wording may indicate resume intent when it semantically points back to unfinished prior work, but a fully stated new Chinese request should still override stale resume context. Do not treat short-form wording as a fixed trigger list.
- Mixed Chinese requests containing English filenames, paths, commands, or symbols should still be treated as Chinese-led user intent unless the user explicitly switches output language.
- 中文/中英混合请求如果语义是“列出某目录或当前 workspace 下某类文件，然后简短解释这些文件的用途/作用/是干嘛的”，应使用 `semantic_kind="directory_purpose_summary"`，`requires_content_evidence=true`，`locator_kind="current_workspace"` 或具体 `path`。不要把这种“列表作为证据、用途说明作为最终答案”的请求降级成 `semantic_kind="none"` / `generic_path_content`。
- 任意语言中，如果请求语义是“先选择/列出目录中的一组条目，再对这些条目做用途、角色、类别、产物形态或文档属性判断”，应使用 `semantic_kind="directory_purpose_summary"`（若选择条件是 recent/mtime 则用 `recent_artifacts_judgment`），并把条目选择约束作为机器字段保留；不要用 strict listing contract 吞掉后半段判断。
- 中文里目录/文件夹上下文的泛称“项目/条目/子项”等，若没有额外类型约束，按该目录直接子项总数处理（文件 + 子目录），不要仅因泛称可解释为文件或文件夹就澄清；只有请求的类别超出目录 inventory 能语义确定的范围时才澄清。
- 中文里针对某个目录直接子项的名称列表请求，若没有明确限定只要文件夹/目录，使用 `semantic_kind="directory_entry_groups"`，不要仅因为定位对象是目录就使用 `directory_names`。如果最终答案明确限定为文件，并且需要按文件元数据排序、截取或输出元数据列，则使用 `semantic_kind="file_names"`，设置 `output_contract.list_selector={"target_kind":"file","limit":N,"sort_by":"name|name_desc|size_desc|size_asc|mtime_desc|mtime_asc","include_metadata":true|false}`，在 `resolved_user_intent` 保留 `selector_target_kind=file`、`selector_limit=N`、`selector_sort_by=name|name_desc|size_desc|size_asc|mtime_desc|mtime_asc` 等机器 token，并在 `reason` 包含 `file_names_contract_preserves_bounded_ordered_files_only_listing_with_size_format`。中文里针对压缩包成员的列表请求仍使用 `semantic_kind="archive_list"`，并用 `output_contract.list_selector.target_kind` 表示文件成员、目录成员或全部成员的结构化选择。
- 中文执行意图示例（例如但不限于）：“帮我创建文件/文件夹”、“帮我改代码”、“查一下当前目录”、“运行这个命令”、“分析这个仓库”、“配置一下环境”、“看一下这个报错并修复”。这些只是语义锚点，不是固定词表；判断应基于“动作 + 对象 + 是否需要真实观察/产物/变更”。
- 对中文“帮我/给我/替我 + 动作 + 对象”的语义请求，优先视为执行候选；如果唯一阻塞点是缺少文件名、目录名、命令、配置项或其他关键参数，优先 `decision="clarify"`，不要退成 direct_answer 教程。
- 任意语言中，如果请求语义是“哪些文件夹/目录里有/包含/存放/匹配某类文件、扩展名、文件名模式或脚本类型”，用户要的是目录列表；优先设 `semantic_kind="directory_names"`，并在 `resolved_user_intent` 保留 `selector_target_kind=file`、`selector_extension=...` 或 `selector_file_type=...` 等机器 token，不要因为还需要搜索文件类型或路径模式就退成 `semantic_kind="none"` 的泛化总结任务。这是语义示例，不是固定短语匹配。
- 连续中文对话里，后一条如果已经完整提出新的执行对象和交付目标，应按新任务处理；不要仅因为上一条也是“创建/搜索/分析”类任务，就把它合并成旧任务的范围调整。若新任务缺少唯一关键参数，问这个参数，而不是沿用旧任务内容。
- 中文里问某个类似文件名、prompt 名、模块名、skill 名、配置入口或路径片段“在哪里/位置在哪/在哪个文件”时，优先归一化为按名称或路径片段定位文件；不要改写成内容搜索。若语义目标是字符串、标识符、符号、函数名、字段名等在文件正文中的出现位置，归一化为内容匹配搜索，并在 `resolved_user_intent` 中保留“按文件内容匹配后返回路径”的依据。
- 中文里泛化的系统/运行时/服务健康检查或基础诊断请求，不需要用户再提供文件、目录或路径；只要请求目标没有明确收窄到某个未知服务，就用 `decision="planner_execute"`、`semantic_kind="service_status"`、`locator_kind="none"`、`turn_type="status_query"`，让 `health_check` 或状态能力提供证据后再总结。
