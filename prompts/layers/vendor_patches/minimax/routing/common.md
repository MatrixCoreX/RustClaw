- Do not use `workspace_project_summary` for repository/workspace-wide extension searches or representative matching path lists. Use `file_paths` unless the requested final answer is a project-level explanation from broader workspace evidence.
- For recent-file listings plus any type/category/purpose/use/role judgment about those recent entries, use `semantic_kind="recent_artifacts_judgment"` and preserve both the listing requirement and the judgment requirement in `resolved_user_intent`. This includes judgments such as document vs artifact, log vs test residue, generated output vs formal deliverable, what selected recent files are mainly for, or other domain-specific categories.

- For exact or constrained final outputs, change only the contract fields. Hidden-entry existence checks use `response_shape="strict"`, `delivery_required=false`, `delivery_intent="none"`, and `semantic_kind="hidden_entries_check"`; file-only names use `response_shape="strict"`, `delivery_required=false`, and `semantic_kind="file_names"`; direct directory child inventory, including names where files and directories may both be valid answers, uses `response_shape="strict"`, `delivery_required=false`, and `semantic_kind="directory_entry_groups"`; exact bounded file/log line slices use `response_shape="strict"`, `delivery_required=false`, and `semantic_kind="raw_command_output"`; exact content-match lines/snippets use `response_shape="strict"` and `semantic_kind="content_presence_check"`; current path only uses `response_shape="scalar"`, `delivery_required=false`, and `semantic_kind="scalar_path_only"`.
- For local filesystem mutations where the final answer should report the observed action result rather than deliver a `FILE:` token, use `response_shape="one_sentence"`, `delivery_required=false`, `delivery_intent="none"`, `semantic_kind="filesystem_mutation_result"`, and `requires_content_evidence=true`. Preserve the target path in `locator_hint` when available. This contract takes priority over `command_output_summary` when the requested answer is a structured success/failure summary of filesystem lifecycle steps.
- For create/save/write requests where the final answer should be only the saved file path in chat, use `response_shape="scalar"`, `delivery_required=false`, `delivery_intent="none"`, `semantic_kind="generated_file_path_report"`, and `requires_content_evidence=true`; do not use `generated_file_delivery` or `FILE:<path>` for this path-report contract.
- Bounded or ordered direct directory child inventory remains `semantic_kind="directory_entry_groups"` when the final answer should include multiple entry names plus type/kind, even if the ordering is by recency or modification time. Keep the order/count constraint in `resolved_user_intent`; do not downgrade this to `semantic_kind="none"`, `quantity_comparison`, or a generic workspace/tree overview. If the requested final answer is restricted to files and depends on file metadata ordering or a file metadata column, use `semantic_kind="file_names"` instead, set `output_contract.list_selector` with target_kind=file plus requested limit/sort/metadata fields and `include_metadata=true`, carry the selector machine tokens in `resolved_user_intent`, and include `file_names_contract_preserves_bounded_ordered_files_only_listing_with_size_format` in `reason` for size-ranked or size-column file lists.
- For directory, filesystem, search-result, or match count requests, use `semantic_kind="scalar_count"` and a scalar response when the final answer is just the count. Do not use `file_names` or `directory_entry_groups` for a pure count request; list/group semantics are only for final answers that include names or grouped entries.
- For scalar count execution plans, use `fs_basic.count_entries` directly and encode object filters as machine args (`dirs_only`, `files_only`, `ext_filter`, `include_hidden`). Do not rely on `list_dir` plus `filter_logic` or prose post-processing to compute the final number.
- Names-only file listing uses `semantic_kind="file_names"` when the final answer is restricted to file entries. Direct child names under a directory use `semantic_kind="directory_entry_groups"` only when files and subdirectories are both valid final entries. For an untyped direct-child inventory of one concrete directory, keep the target kind mixed (`files + directories`) and do not set `files_only=true`; when the requested noun class is file entries rather than generic directory entries, choose `file_names`, preserve `output_contract.list_selector={"target_kind":"file"}` (or matching machine token `selector_target_kind=file`), and make the later plan use `files_only=true`. If the same turn asks for a brief purpose/use explanation or grounded category/style judgment of those names, switch to `semantic_kind="directory_purpose_summary"` so the final answer may include model language grounded in the observed listing.
- Use `scalar_path_only` only for a final answer with exactly one path. For bounded search, conditional fallback search, representative extension search, candidate lookup, or "return found paths" tasks that may produce multiple paths, use `response_shape="strict"` and `semantic_kind="file_paths"`.
- If the request supplies a directory plus a search token and asks to find matching entries inside that directory, keep the directory as `locator_hint` and the token/search criterion in `resolved_user_intent`; do not collapse the contract to checking whether the directory path itself exists.
- For fixed-line or exact-format inline answers containing multiple observed values plus a conclusion, use `response_shape="strict"` and preserve the exact final format in `resolved_user_intent`. Do not invent aliases like `one-line string`, translated enum text, `concatenated_comparison`, or `content_comparison`.
- For equality comparison of scalar field values that still need fresh file/tool observations, use `decision="planner_execute"`, `requires_content_evidence=true`, `delivery_required=false`, `delivery_intent="none"`, and `semantic_kind="recent_scalar_equality_check"`. Keep the one-line/exact-format constraint in `response_shape="strict"` when the user requested it. Preserve known structured dot-paths as machine tokens in `resolved_user_intent` (for example `workspace.package.version`), and treat filename extensions such as the `toml` in `Cargo.toml` as locator syntax rather than field selectors.
- For comparison or classification of prose excerpts/opening sections by audience, purpose, document role, or content type, use `semantic_kind="excerpt_kind_judgment"` with content evidence. Do not route these as scalar equality checks.
- If the user asks whether hidden files/hidden entries/dot-prefixed entries exist and asks to include matching entries, the final answer is not a general directory listing and not a plain file-name listing. The only valid contract is `response_shape="strict"` plus `semantic_kind="hidden_entries_check"` and `output_contract.list_selector.include_hidden=true`. Keep this contract even when the final answer also asks for a brief purpose/use/explanation of such entries; do not route it as `file_names`, `existence_with_path`, or `existence_with_path_summary` because the observed target is the dot-prefixed entry set, not ordinary files or the directory path itself.
- MiniMax schema example: a request whose semantic meaning is “check whether the current directory has hidden files/entries and briefly explain what such entries are for” must output `decision="planner_execute"`, `locator_kind="current_workspace"`, `requires_content_evidence=true`, `semantic_kind="hidden_entries_check"`, `response_shape="strict"`, and `output_contract.list_selector={"target_kind":"any","include_hidden":true}`. Do not set `semantic_kind="file_names"` and do not set `include_hidden=false`.
- For Docker container-management readiness/capability checks, use `semantic_kind="docker_container_lifecycle"` even when the user asks for availability rather than an immediate start/stop/restart mutation. Reserve `semantic_kind="service_status"` for ordinary host service/process status.
